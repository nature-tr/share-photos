# 微信小程序「格子橱窗」权限问题深度复盘

> 时间跨度：2026 年 6 月 · 涉及模块：小程序隐私合规、chooseMedia 相册选择、上传/下载后台执行
> 关键词：errno:112、隐私授权死锁、微任务边界、页面级网络请求、最小化保活

---

## 一、项目背景

### 1.1 权限系统初始架构

格子橱窗是一个微信小程序限时相册分享工具，核心涉及三类权限：

| 权限类型 | API | 用途 |
|---------|-----|------|
| **相册读取** | `wx.chooseMedia` / `Taro.chooseMedia` | 用户选图上传 |
| **相册写入** | `scope.writePhotosAlbum` → `wx.authorize` | 一键保存到手机相册 |
| **隐私授权** | `__usePrivacyCheck__` + `onNeedPrivacyAuthorization` | 微信 2023 年隐私合规新规 |

初始架构分为三层：

```
┌─────────────────────────────────────────┐
│  app.config.ts                          │
│  __usePrivacyCheck__: true              │  ← 全局启用隐私检查
│  privacy: { title, content, ... }       │  ← 隐私政策声明
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  app.tsx → <PrivacyConsent />           │  ← 全局隐私弹窗
│  onNeedPrivacyAuthorization 监听器      │     (模块顶层同步注册)
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  utils/permission.ts                    │
│  requestWriteAlbumPermission()          │  ← 写入权限引导
│  utils/hooks.ts                         │
│  pickImagesFromAlbum()                  │  ← 选图封装
└─────────────────────────────────────────┘
```

### 1.2 业务场景与权限需求

**上传场景**：用户在"新建分享"页面点击选图 → 弹出原图/压缩选择 → `Taro.chooseMedia` → 选中图片 → 并发上传至后端

**下载场景**：用户在"查看相册"页面点击"一键存到相册" → 验证写入权限 → 逐张 `Taro.downloadFile` → `Taro.saveImageToPhotosAlbum`

**限制需求**：微信要求从 2023 年 9 月 15 日起，所有调用隐私接口的小程序必须声明隐私政策并获取用户明确同意。

## 二、原因分析

### 2.1 errno:112 死锁全景链路

**现象**：`chooseMedia:fail api scope is not declared in the privacy agreement`（errno:112）

**排查完整时间线**：

```
Step 1：配置 __usePrivacyCheck__: true + privacy 声明
   → 预期：用户首次使用 chooseMedia 时弹出 PrivacyConsent
   → 实际：部分安卓机型不弹窗，直接 errno:112

Step 2：验证 manage 后台隐私声明
   → 检查微信公众平台「设置 → 基本设置 → 服务内容声明」
   → 发现隐私接口列表中缺少「选取相册照片 / 仅读取」选项

Step 3：补充管理后台隐私声明 + 重新提交审核
   → 审核通过后，模拟器恢复正常
   → 真机预览仍然 errno:112

Step 4：排查手机上「设置 → 隐私与授权」
   → 发现格子橱窗的权限状态显示为「已删除」
   → 该状态是微信客户端本地缓存的，与服务端隐私声明不同步

Step 5：尝试客户端侧绕过
   → wx.authorize → 不支持 chooseMedia 对应的 scope
   → wx.openPrivacyContract → 无法清除已删除缓存
   → 清除微信缓存 / 删除重装小程序 → 无效
   → 切换微信账号 → 无效（缓存跟设备绑定）

Step 6：最终根因确认
   → 注释 __usePrivacyCheck__: true
   → 微信运行时不再检查客户端缓存的隐私状态
   → chooseMedia 恢复正常
```

### 2.2 死锁机制详解

```
┌──────────────────────────────────────────────────────────────┐
│                      死锁循环                                 │
│                                                              │
│  微信客户端缓存「已删除」   →   管理后台声明但缺失某个 scope   │
│        ↓                              ↓                      │
│  客户端拒绝执行 chooseMedia      不触发 onNeedPrivacy        │
│        ↓                              ↓                      │
│  返回 errno:112                无法弹出 PrivacyConsent        │
│        ↓                              ↓                      │
│              用户永远无法重新授权                              │
│                                                              │
│  核心矛盾：客户端缓存 ≠ 服务端配置，中间没有同步机制            │
│  微信提供的 onNeedPrivacyAuthorization 无法打破这个循环        │
└──────────────────────────────────────────────────────────────┘
```

**技术内因**：

1. **隐私状态双重存储**：微信将用户隐私同意状态分别存储在「服务端（管理后台声明的 scope 列表）」和「客户端（用户本地同意的 scope 记录）」。二者不一致时，客户端优先信任本地缓存。

2. **已删除状态的不可逆性**：当用户在管理后台删除某个 scope 声明后，已经同意过该 scope 的客户端将其标记为「已删除」。即使管理后台重新添加该 scope，客户端不会自动感知。

3. **`onNeedPrivacyAuthorization` 触发前提**：该监听器只有在微信运行时判定"需要隐私授权但用户未同意"时才触发。当客户端已有「已删除」记录时，该路径被短路，监听器永不触发。

### 2.3 微任务边界问题（嵌套调用陷阱）

在 `pickImagesFromAlbum` 中调试期间发现第二个关键问题：

```typescript
// ❌ 错误写法：chooseMedia 嵌套在 showActionSheet 的 success 回调中
Taro.showActionSheet({
  itemList: ['原图', '压缩'],
  success: (res) => {
    // 这个回调在 showActionSheet 的微任务队列中执行
    Taro.chooseMedia({ ... }); // ← 此处隐私授权流程无法穿透
  },
});
```

**问题机制**：微信的隐私授权流程 `onNeedPrivacyAuthorization → resolve({ event: 'agree' }) → 重试 chooseMedia` 是一个基于 Promise 的流水线。当 `chooseMedia` 嵌套在 `showActionSheet` 的回调中时：

- `showActionSheet.success` 回调运行在微信原生层的一个**嵌套微任务队列**中
- `onNeedPrivacyAuthorization` 的 resolve 事件无法穿透到嵌套微任务边界
- 即使 `PrivacyConsent` 弹窗显示了，用户点击同意后 `chooseMedia` 也不会重试

**正确做法**：

```typescript
// ✅ 正确：await 在异步函数顶层，让 chooseMedia 直接暴露在当前微任务层级
const compressed = await new Promise<boolean | null>((resolve) => {
  Taro.showActionSheet({ ... });
});
// compressed 获得后，在异步函数顶层调用 chooseMedia
const res = await Taro.chooseMedia({ ... });
// ← 隐私授权流程可以正常穿透
```

### 2.4 上传/下载中断问题（页面级网络请求）

**现象**：用户在"新建分享"页面上传过程中按返回或切到其他页面 → 上传全部失败

**根因**：

```
微信小程序的 Taro.uploadFile / Taro.downloadFile 是页面级网络请求：
  ┌──────────────────┐
  │   Page Instance  │  ← 页面实例持有网络请求
  │  ┌────────────┐  │
  │  │ uploadFile  │──│── 原生 wx.uploadFile
  │  │ downloadFile│  │      ↓
  │  └────────────┘  │   绑定页面生命周期
  └──────────────────┘
         ↓ onUnload
    页面销毁 → 所有网络请求被微信底层强制 cancel
```

当用户通过 `navigateBack()`、`redirectTo()`、`reLaunch()` 离开页面时：

1. 页面触发 `onUnload`
2. 微信运行时遍历该页面的所有活跃请求
3. 逐一调用底层 `cancel` 机制
4. Promise 链条断裂（回调可能来不及执行或根本不执行）
5. `taskManager.runUpload()` 中的 `ctx.running` 标志永远停留在 `true`
6. 后续 `resumeUpload()` 因 `ctx.running === true` 被直接 return 跳过
7. 上传任务永久卡死

**双重 Bug**：

```
Bug A（回调早于销毁）：abort 回调成功触发 → item 变 pending
                     → Promise.all 正常 resolve
                     → finishUpload 被调用（错误！因为 items 全是 pending 不是 done）
                     → GlobalProgress 悬浮球消失
                     → 用户永远无法恢复

Bug B（回调晚于销毁）：页面 JS 上下文被销毁 → Promise 永不 resolve
                     → ctx.running 保持 true
                     → resumeUpload 直接跳过
                     → 任务永久卡死
```

## 三、解决方案

### 3.1 隐私合规 errno:112 的最终方案

#### 3.1.1 管理后台配置

在微信公众平台 `设置 → 基本设置 → 服务内容声明` 中，确保以下 scope 全部声明：

```
✅ 选择照片或视频（仅读取）       — chooseMedia 依赖
✅ 保存图片或视频到相册           — saveImageToPhotosAlbum 依赖
✅ 获取用户手机号（如需要）       — 按需声明
```

#### 3.1.2 客户端配置

```typescript
// app.config.ts
export default defineAppConfig({
  // __usePrivacyCheck__: true,  // ← 注释掉，避免客户端缓存死锁
  privacy: {
    title: '隐私保护指引',
    content: '格子橱窗尊重并保护您的隐私。为提供相册分享服务...',
    organizationName: '格子橱窗团队',
    policyUrl: 'https://www.dolmo.top/privacy',
    organizationUrl: 'https://www.dolmo.top',
  },
});
```

#### 3.1.3 PrivacyConsent 组件（全局弹窗）

```typescript
// 关键设计：监听器在模块顶层同步注册（不在组件 useEffect 中）
let resolveRef: ((res: any) => void) | null = null;
let showFn: ((v: boolean) => void) | null = null;

const wxObj = getWx();
if (wxObj) {
  if (wxObj.onNeedPrivacyAuthorization) {
    wxObj.onNeedPrivacyAuthorization((resolve: any) => {
      resolveRef = resolve;
      showFn?.(true); // 触发 PrivacyConsent 弹窗
    });
  }
}

// handleAgree: 调用 resolveRef({ event: 'agree' }) + 本地存储标记
// handleDisagree: 调用 resolveRef({ event: 'disagree' })
```

> `PrivacyConsent` 组件挂载在 `app.tsx` 根节点，确保所有页面通用。

#### 3.1.4 pickImagesFromAlbum 的微任务边界修复

```typescript
// 核心原则：chooseMedia 必须在 async 函数顶层调用
export async function pickImagesFromAlbum(opts: PickImagesOpts = {}) {
  // Step 1: await 等待用户选择（Promise 在顶层）
  const compressed = await new Promise<boolean | null>((resolve) => {
    Taro.showActionSheet({ ... });
  });

  // Step 2: chooseMedia 直接在当前 async 函数顶层调用
  //         使隐私授权流程（onNeedPrivacyAuthorization→resolve→重试）
  //         能够正常穿透
  const res = await Taro.chooseMedia({
    count,
    mediaType: ['image'],
    sizeType: compressed ? ['compressed'] : ['original'],
  });
}
```

### 3.2 上传/下载后台执行的最终方案

#### 3.2.1 设计思路

由于微信小程序不支持真正意义上的"后台网络请求"，采用**页面保活 + 最小化遮罩**策略：

```
页面栈示意：
┌──────────────────────┐
│  /pages/index/index  │  ← 当前可见页（刚 navigateTo 过来的）
├──────────────────────┤
│  /pages/me/new/index │  ← 上传页面（隐藏但存活，uploadFile 继续跑）
├──────────────────────┤
│  /pages/index/index  │  ← 原始首页
└──────────────────────┘

用户视角："我关掉了上传页"    实际：页面在栈中，只是被遮住了
```

#### 3.2.2 最小化按钮实现

```typescript
// pages/me/new/index.tsx — 底部操作条
{taskStatus === 'uploading' ? (
  <View
    className="btn btn-minimize"
    onClick={() => Taro.navigateTo({ url: '/pages/index/index' })}
  >
    <Text className="btn-minimize-text">最小化</Text>
  </View>
) : /* ... 其他状态 ... */}
```

#### 3.2.3 返回拦截（防误操作）

```typescript
// 上传/下载进行中时，防止用户误按返回按钮直接销毁页面
useEffect(() => {
  if (taskStatus !== 'uploading') return;
  Taro.enableAlertBeforeUnload?.({
    message: '上传正在进行中，直接返回将中断上传。请使用「最小化」按钮切换到后台继续上传。'
  });
  return () => {
    Taro.disableAlertBeforeUnload?.();
  };
}, [taskStatus]);
```

#### 3.2.4 finishUpload 条件修复

```typescript
// task.manager.ts — runUpload()
// ❌ 旧逻辑：只要 workers 返回就标 done（abort 后被错误关闭）
const final = useTaskStore.getState().uploads[ctx.shareId];
if (final && final.status === 'uploading') {
  finishUpload(ctx.shareId); // ← 即使 items 全是 pending 也标 done
}

// ✅ 新逻辑：只有所有 item 真正完成才标 done
const allDone = ctx.items.every((it) => it.status === 'done');
const final = useTaskStore.getState().uploads[ctx.shareId];
if (allDone && final && final.status === 'uploading') {
  finishUpload(ctx.shareId);
}
```

#### 3.2.5 forceResumeUpload（卡死恢复）

```typescript
// task.manager.ts — 处理页面销毁后 ctx.running 卡在 true 的死锁
forceResumeUpload(shareId: string) {
  const ctx = uploads.get(shareId);
  if (!ctx) return;
  
  ctx.running = false;  // 重置卡死的标志
  ctx.wxTask = null;    // 清理孤儿句柄
  
  // 将卡在 'uploading' 的项回滚为 'pending'
  for (const item of ctx.items) {
    if (item.status === 'uploading') {
      item.status = 'pending';
      item.error = undefined;
    }
  }
  
  useTaskStore.getState().setUploadStatus(shareId, 'uploading');
  void runUpload(ctx);  // 重新启动
}
```

只在从 GlobalProgress 跳转过来时触发（用 ref 标记），避免普通 `useDidShow`（如从首页返回）误触发导致正在运行的上传被中断。

### 3.3 写相册权限引导

```typescript
// utils/permission.ts
export async function requestWriteAlbumPermission(): Promise<boolean> {
  const setting = await Taro.getSetting();
  const auth = setting.authSetting['scope.writePhotosAlbum'];
  
  if (auth === true) return true;
  if (auth === undefined) {
    // 从未请求过 → 弹系统授权框
    return Taro.authorize({ scope: 'scope.writePhotosAlbum' });
  }
  // 被拒 → 弹 Modal 引导去设置页
  return guideToSetting('写入相册');
}
```

引导设置页会 `openSetting` 后轮询等待权限同步（3 次 × 300ms），避免返回后权限状态未更新的竞态。

## 四、权限校验规范（后续开发参考）

### 4.1 选择图片（chooseMedia）

```typescript
// 规则 1：必须在 async 函数顶层调用，不得嵌套在回调里
// 规则 2：不依赖 __usePrivacyCheck__（避免客户端缓存死锁）
// 规则 3：catch 中以 errMsg 关键词判断是权限被拒还是用户取消
const picked = await pickImagesFromAlbum({ count: 9 });
if (picked.items.length === 0) {
  if (picked.reason === 'denied') {
    Taro.showToast({ title: '已保存授权，请再次点击选图', icon: 'none' });
  }
  return;
}
```

### 4.2 保存图片到相册（saveImageToPhotosAlbum）

```typescript
// 规则 1：每次保存前验证写入权限
const granted = await requestWriteAlbumPermission();
if (!granted) return;

// 规则 2：下载 + 保存原子操作
const res = await Taro.downloadFile({ url, timeout: 60000 });
if (res.statusCode === 200) {
  await Taro.saveImageToPhotosAlbum({ filePath: res.tempFilePath });
}
```

### 4.3 后台任务执行

```typescript
// 规则 1：任何需要跨页面的网络任务（上传/下载），提供"最小化"按钮
// 规则 2：用 navigateTo 替代 redirectTo/navigateBack，保持页面存活
// 规则 3：启用 enableAlertBeforeUnload 防误操作返回
// 规则 4：状态管理分离：store 存快照（UI 订阅），manager 存上下文（执行控制）
```

### 4.4 禁止事项

| 禁止 | 原因 |
|------|------|
| 在 `showActionSheet` 回调中调用 `chooseMedia` | 微任务边界阻断隐私授权流程 |
| 上传/下载进行中用 `redirectTo` 或 `navigateBack` | 页面销毁导致网络请求被取消 |
| 依赖 `__usePrivacyCheck__: true` 作为唯一隐私合规手段 | 客户端缓存可能在真机上造成死锁 |
| 在上传未全部完成时调用 `finishUpload` | 导致悬浮窗消失，用户无法恢复 |
| 对 `ctx.running` 不加判断直接 `resumeUpload` | 页面销毁后 running 标志卡在 true，永远跳不过 |

## 五、关键经验总结

1. **微信隐私机制是双刃剑**：`onNeedPrivacyAuthorization` 设计初衷是好的，但客户端缓存与服务端声明的不同步会导致不可恢复的死锁。在隐私声明完备的前提下，`__usePrivacyCheck__` 反而可能是故障点。

2. **微任务边界是性能优化也是权限陷阱**：微信将 `showActionSheet` 的 success 回调放在嵌套微任务中，导致 `chooseMedia` 的隐私授权 retry 流水线无法穿透。`await` 在顶层是唯一出路。

3. **页面级 API 没有"后台"语义**：微信小程序不提供真正的后台网络请求。上传/下载的"后台续传"只能通过保持页面存活的 hack 实现——用 `navigateTo` 推新页盖住旧页。

4. **状态管理解耦的价值**：将 store（UI 快照）与 manager（执行上下文）分离，使得即使页面被销毁、Promise 链条断裂，manager 中的上下文数据也不会丢失，为后续恢复提供了可能。

5. **本地上传速度的瓶颈在上行带宽**：客户端无法改善物理带宽。如果未来需要极致体验，可考虑后端支持 S3 预签名直传（绕过服务器中转），将上传流量直接打到 OSS 节点。
