## 3. 关键实现要点

### 3.1 JWT 双 Token + Cookie

- Access token：`@fastify/jwt`，有效期 15min，前端存内存（Pinia）
- Refresh token：自定义随机串（`<jti>.<base64url>`），SHA-256 哈希存 DB；下发到 `httpOnly + SameSite=Lax` cookie，path `/api/auth`
- Rotate：每次 `/refresh` 签新 token + 吊销旧 token
- 重放检测：被吊销的 refresh 再次使用 → 吊销该用户全部 refresh（强制下线）
- 前端 `client.ts` 在 401 + 自动 refresh 一次后重试原请求；并发请求共用同一个刷新 Promise

### 3.2 上传链路

1. 前端选图 → 选 TTL → 选「原图 / 压缩」
2. `POST /api/shares` 拿到 shareId / shareCode
3. 文件并发上传（限 3 路）：
   - `compressed` 模式：`browser-image-compression`（长边 4000，JPEG q0.92）
   - `original` 模式：直接走 multipart
4. 后端 `@fastify/multipart` 流式落盘到 `storage/originals/<shareId>/<photoId>.<ext>`
5. sharp 同步生成 thumb / medium，写入 photos 表
6. 任一步失败：清理该文件、给出错误码

### 3.3 查看链路

- `GET /api/v/:code` → 校验 `active` + 未过期 + 未结束 → 返回照片元
- 网格用 `<img loading="lazy" src="/api/v/:code/photos/:id/thumb">`
- 大图：PhotoSwipe 直接打开 original URL（标准做法）；如果未来要先 medium 后 original，把 PhotoSwipe 的 `data-pswp-src` 换成 medium，自定义 UI 加「查看/下载原图」即可
- 全量 zip：`archiver({ store: true })` pipe 到 `reply.raw`，文件名冲突时追加 `<photoId 前 6 位>` 后缀

### 3.4 过期清理

- `node-cron` 每分钟扫一次 `cleanup.runCleanupOnce()`
- 启动时立即跑一次，避免重启窗口期残留
- 顺序：删文件目录 → 删 photos 行 → 更新 share.status='cleaned'
- 单轮上限 50 个 share，避免长尾任务卡死
- 任一 share 失败仅记 error 日志，不抛出影响整体调度

## 4. 数据流与错误处理

### 4.1 响应格式

成功：`{ data: <T> }` ｜ 失败：`{ error: { code, message, details? } }`

### 4.2 错误码常量

定义在 `shared/src/constants/error-codes.ts`，前后端共用（可由前端针对性 UI 提示）。

### 4.3 错误流转

- 业务错误：抛 `BizError(code, message, statusCode)` → 错误处理插件转 `{error: ...}`
- Zod 校验错误：自动转 `VALIDATION_FAILED`
- 上传过大：`@fastify/multipart` 抛 `FST_REQ_FILE_TOO_LARGE` → 转 `FILE_TOO_LARGE`
- 限流：429 → `RATE_LIMITED`
- 兜底：未识别异常 → 500 + `INTERNAL_ERROR`，原始堆栈仅写日志

## 6. 踩坑记录（部署/重装环境时参考）

### 6.1 better-sqlite3 prebuild 与 Node ABI

- 项目锁 `better-sqlite3@11.5.0`，darwin-arm64 同时含 Node v115/v127/v131 的 prebuild（Node 20/22/24 都覆盖）。
- **切换 Node 版本后必须重装 native 模块**，否则会报：
  ```
  NODE_MODULE_VERSION 115 ≠ 127  (Node 20 装的二进制 vs Node 22 运行)
  ```
  解决：
  ```bash
  rm -rf node_modules/.pnpm/better-sqlite3* node_modules/better-sqlite3 backend/node_modules/better-sqlite3
  npm_config_cache="$PWD/.npm-cache" pnpm install
  ```

### 6.2 ~/.npm/_prebuilds/ 权限

部分 macOS 上该目录被 root 占用，导致 `prebuild-install` 写入失败回退到 native 编译，再因为只有 CLT 没有完整 Xcode 而失败。

固定方案：使用项目级缓存

```bash
npm_config_cache="$PWD/.npm-cache" pnpm install
```

### 6.3 drizzle-kit push 是交互式

不便自动化。本项目用 `drizzle-kit generate` 生成迁移，结合 `drizzle-orm/better-sqlite3/migrator` 在启动时和 `pnpm db:push` 命令中应用。修改 schema 流程：

```bash
# 1. 改 backend/src/db/schema.ts
# 2. 生成迁移
cd backend && pnpm db:generate
# 3. 应用迁移（或直接重启后端，server.ts 启动时也会自动应用）
pnpm db:push
```

### 6.4 Vue 模板里 inline arrow function 不能写 TS 类型

Vue 模板 SFC 里 `@click="(d: { value: number }) => ..."` 会被 vue-tsc 当 JS 解析报错。
解决：把回调抽到 `<script setup>` 里命名函数，模板里只写 `@click="onClick"`。

## 7. 扩展指引

### 7.1 增加一张表 / 一个字段

1. 改 `backend/src/db/schema.ts`
2. `cd backend && pnpm db:generate` 生成迁移
3. 重启后端（`server.ts` 自动应用）

### 7.2 增加一个新的 API 模块

```
backend/src/modules/<feat>/
  <feat>.routes.ts   # 路由声明
  <feat>.service.ts  # 业务逻辑
  (<feat>.repository.ts 如有需要)
```

在 `app.ts` 注册：`await app.register(featRoutes, { prefix: '/api/<feat>' })`

### 7.3 前端新增一个页面

1. `frontend/src/pages/<feat>/<Page>.vue`
2. `frontend/src/router/index.ts` 注册路由（按需加 `meta.requiresAuth`）

### 7.4 共享类型

放到 `shared/src/dto/` 或 `shared/src/constants/`，导出后前后端按 `@photo/shared` 引用。

## 8. 移动端原生 App（Expo + React Native）

### 8.1 概述

`app/` 是基于 Expo SDK 54 + React Native 0.81 的原生移动端 App，独立于 Web 前端。Android 端已构建并上线 APK 下载。

**核心能力**：
- 注册/登录（与后端共享 API）
- 凭码访问相册 + 图片网格浏览
- 扫码进入（相机扫描二维码或分享码）
- 创建分享 + 从手机相册选图上传
- 分享管理（续期/结束）
- 保存图片到手机相册（原生权限）

**与原 Web 前端的关系**：共享同一套后端 API 和 `@photo/shared` 类型/常量。两者功能等价，移动端额外享有原生相机/相册/存储权限。

### 8.2 技术栈

| 项 | 选型 |
| --- | --- |
| 框架 | Expo SDK 54 + React Native 0.81 |
| 路由 | expo-router（文件路由） |
| 状态管理 | Zustand |
| 图片处理 | expo-image + expo-image-picker + expo-media-library |
| 相机扫码 | expo-camera |
| JS 引擎 | Hermes |
| 架构 | Android: arm64-v8a only（覆盖 99%+ 设备） |

### 8.3 本地开发

```bash
cd app

# 1. 先编译 shared 包（metro bundler 需要 dist/）
pnpm -C ../shared build

# 2. 启动 Expo dev server
pnpm start

# 然后按 a 开 Android 模拟器，或扫码 expo go 运行
```

### 8.4 本地打包 Android APK

**前置要求**：
- JDK 17（路径通过 `JAVA_HOME` 指定）
- Android SDK CLI tools + platform-tools + build-tools 36

```bash
cd /Users/chengtianran/CodeBuddy/photo/app

# 一条命令完成：编译 shared → prebuild → assembleRelease
pnpm -C ../shared build && \
npx expo prebuild --platform android --no-install && \
cd android && \
./gradlew :app:assembleRelease --no-daemon --max-workers=2 && \
cp app/build/outputs/apk/release/app-release.apk \
   /Users/chengtianran/CodeBuddy/photo/app/dolmo-photo-v0.1.0.apk && \
ls -lh /Users/chengtianran/CodeBuddy/photo/app/dolmo-photo-v0.1.0.apk
```

**关键参数说明**：
- `--no-daemon`：单次构建，build 完 JVM 立即退出释放内存；防止多次构建后 daemon 累计吃内存
- `--max-workers=2`：限制 Gradle 任务并发数，避免 CMake 编译 hermes 等 native 时 OOM
- 产物默认用 debug keystore 签名，可直接安装

**Android 体积优化**：
- 仅编译 `arm64-v8a` ABI（通过 `with-arm64-only` Expo plugin，在 gradle.properties 中写 `reactNativeArchitectures=arm64-v8a`）
- 效果：APK 从约 107 MB 降至约 42 MB
- 未启用 R8/资源压缩（尝试开启后出现 R8 与 shrinkResources 配置冲突，留待后续）

### 8.5 图标管理

所有应用图标使用同一张品牌 PNG，放在 `app/assets/`：

| 文件 | 用途 | 说明 |
| --- | --- | --- |
| `icon.png` | iOS / 旧 Android 桌面图标 | 原图 1372×1410 |
| `adaptive-icon.png` | Android 自适应图标前景 | 1024×1024，原图缩放至中间 66%，外圈透明 padding，防系统 mask 裁切 |
| `adaptive-icon-bg.png` | Android 自适应图标背景 | 1024×1024 全透明 PNG |
| `splash.png` | 启动屏图标 | 原图，白底 `#FFFFFF` 居中 contain 显示 |

### 8.6 上传到服务器

```bash
# 把 APK 传到 www.dolmo.top 的 frontend/dist/ 目录，并维护 latest 软链
bash scripts/upload-apk.sh
```

上传后两个下载链接：
- `https://www.dolmo.top/dolmo-photo-v0.1.0.apk`（永久）
- `https://www.dolmo.top/dolmo-photo.apk`（latest，每次上传自动指向最新版）

> ⚠️ 注意：`deploy.sh frontend` 用 `rsync --delete`，会把远端 dist/ 目录下的 APK 清除。每次部署完前端后需重新跑一次 `upload-apk.sh`。

### 8.7 修改图标 / 品牌素材后重新打包

1. 替换 `app/assets/icon.png`（如需调整 adaptive-icon 居中比例，改 `assets/adaptive-icon.png`）
2. 修改首页品牌区 logo：编辑 `app/app/index.tsx`（`<Image source={require('../assets/icon.png')}>`）
3. 重新 prebuild + assembleRelease
4. 运行 `upload-apk.sh` 推到服务器
