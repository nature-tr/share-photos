### 1.4 文件存储

- MVP：**本地文件系统**，目录约定：
  ```
  storage/
    originals/<shareId>/<photoId>.<ext>     # 原图（用户上传时的原始格式）
    previews/<shareId>/<photoId>.jpg        # 缩略图：短边 ~400, JPEG q80
    mediums/<shareId>/<photoId>.jpg         # 中等预览图：长边 ~1600, JPEG q85
  ```
- 抽象一层 `StorageDriver` 接口，未来可平滑切换到 S3/OSS/COS。

## 2. 关于 Docker（暂缓）

用户当前**无 Docker 经验**，经讨论决定：

- **MVP 开发期间不上 Docker**，专注业务，本地 `pnpm dev` 启动即可。
- 待 MVP 完成、准备部署到自有服务器时再引入：
  - 编写 `Dockerfile`（前后端分别构建）
  - 编写 `docker-compose.yml`（一键起服务）
  - 同步提供「项目专属 Docker 入门 + 部署指南」文档
- 这样学习曲线最平滑：先熟悉项目结构，再"装箱"。

## 4. 默认约定（实施版）

- **Node 版本**：≥ 20 LTS（开发用 20.18）
- **包管理**：pnpm 9（通过根 `package.json` 的 `packageManager` 字段锁定）
- **TypeScript**：`strict: true` + `noUncheckedIndexedAccess`（shared 严格，前后端按需放宽）
- **代码风格**：约定 ESLint + Prettier，MVP 阶段未配置具体规则文件，后续按需补
- **接口风格**：RESTful + JSON
- **响应格式**：成功 `{ data: <T> }`，失败 `{ error: { code, message, details? } }`
- **时间格式**：API 与 DB 一律使用毫秒级 UNIX 时间戳（数字）
- **ID 生成**：
  - 内部 ID 用 `nanoid` 16 字符（字母数字混合）
  - 分享码用 `nanoid` + 自定义字符集（去除易混淆的 0/O/1/l/I），8 位

## 5. 实施备注（实际跑起来踩到的坑）

> 与未来部署/重装环境有关，记录在此免得重复踩。

### 5.1 better-sqlite3 版本锁定

`better-sqlite3` 的 prebuild 发布并不连续（部分版本只发 Electron prebuild）。
本项目锁到 **`11.5.0`**，该版本 darwin-arm64 同时提供 Node v115 / v127 / v131 的 prebuild，覆盖 Node 20 / 22 / 24，避免本地编译。

> **切换 Node 版本（如 Node 20 → 22）后必须重装 native 模块**：之前 prebuild-install 按当时的 ABI 拉过一份 .node 文件放在 pnpm store 里，对新 Node 无效，会报 `NODE_MODULE_VERSION` 不匹配。
>
> ```bash
> rm -rf node_modules/.pnpm/better-sqlite3* node_modules/better-sqlite3 backend/node_modules/better-sqlite3
> npm_config_cache="$PWD/.npm-cache" pnpm install
> ```

### 5.2 npm prebuild 缓存权限

部分机器上 `~/.npm/_prebuilds/` 被 root 占用导致 `prebuild-install` 写入失败，进而触发 native 编译走 gyp 失败。
解决：使用项目级 npm 缓存：
```bash
npm_config_cache="$PWD/.npm-cache" pnpm install
```
README 中已固化这一安装方式。

### 5.3 数据库初始化方式

`drizzle-kit push` 是交互式命令（不便自动化），项目改用 **生成迁移 + 启动时自动应用** 的模式：

- `pnpm db:generate` 生成迁移 SQL（schema 变更后执行）
- `pnpm db:push` 触发自定义脚本应用迁移（非交互）
- 后端 `server.ts` 启动时自动调用 `runMigrations()`
