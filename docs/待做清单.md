# 待做清单

> 2026-06-08 整理，来自全项目代码审查的剩余优化建议，改动较大，单独立项。

---

## 1. 统一 shared 包的构建与引用方式

**现状**：三端对 `@photo/shared` 的引用方式不一致：

| 端 | 引用方式 | 实际指向 |
|---|---|---|
| backend | `@photo/shared` | `../shared/src/*`（tsconfig paths） |
| frontend | `@photo/shared` | `../shared/src/index.ts`（vite alias + tsconfig paths） |
| miniapp | `@photo/shared` + 子路径 | `shared/dist`（webpack chain alias） |

**问题**：
- 改 dto 后需要手动 `pnpm run build` 到 `shared/dist`，否则 miniapp 看不到
- 三端写法不统一，容易漏更新 dist
- IDE 跳转体验不一致

**建议**：统一全走 `src`（dev 体验好）或全走 `dist`（产物边界清晰），选一个方向。

---

## 2. API Client 公共逻辑抽到 shared

**现状**：三端各自实现了一套 token 刷新 + 401 重试 + 错误格式化的逻辑：

- `miniapp/src/api/client.ts`
- `frontend/src/api/client.ts`
- `backend`（服务端无 client）

**建议**：把 transport-agnostic 部分（token 生命周期管理、错误码映射、401 重试策略）抽到 `shared/api/client.ts`，各端只注入具体的请求函数（`fetch` / `Taro.request` / `axios`）。

---

## 3. auth.store 命名重构：useAuth → authStore + useAuthStore

**现状**：`useAuth` 既被用于 React hook 订阅，又被大量以 `useAuth.getState()/setState()` 方式绕过 React 调用（如 `task.manager.ts`、`client.ts`）。

**问题**：
- 命名 `useAuth` 暗示它是 React hook，但 80% 的调用是纯 JS 调用
- 同一 store 承担两种形态，语义不清

**建议**：拆分双形态：
```ts
// auth.store.ts
export const authStore = create<AuthState>(...);   // 纯 JS 用
export const useAuthStore = authStore;              // React hook 用（别名/默认导出）
```
组件内用 `useAuthStore(s => s.user)`，manager 里用 `authStore.getState().getAccessToken()`。
