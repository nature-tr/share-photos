## 1. 整体架构

### 1.1 部署拓扑（MVP）

```
┌─────────────────────────┐         HTTPS / HTTP
│        Browser          │  ───────────────────────────┐
│  (Vue 3 SPA + Pinia)    │                              │
└─────────────────────────┘                              │
┌─────────────────────────┐                              │
│    Native Mobile App    │                              ▼
│  (Expo SDK 54 + RN 0.81)│              ┌───────────────────────────┐
│    APK install / 扫码   │──────────────│      Fastify Server       │
└─────────────────────────┘  HTTPS/HTTP  │   (Node.js + TypeScript)  │
                                         │ ┌───────────────────────┐ │
                                         │ │  Routes / Controllers │ │
                                         │ │  Services             │ │
                                         │ │  Repositories (Drizzle)│ │
                                         │ │  Storage Driver        │ │
                                         │ │  Cron (cleanup)        │ │
                                         │ └───────────────────────┘ │
                                         └───────────────────────────┘
                                                 │             │
                                                 ▼             ▼
                                      ┌───────────────┐  ┌──────────────┐
                                      │  SQLite (.db) │  │ FileSystem   │
                                      │  data/app.db  │  │  storage/... │
                                      └───────────────┘  └──────────────┘
```

- **MVP 阶段前后端同机部署**：Fastify 既提供 API，也直接负责文件流式返回（缩略图/原图/zip）。
- **不引入** Nginx / Redis / 对象存储，保持最小化。
- 后续如需扩容，从「文件系统 → 对象存储」「SQLite → PostgreSQL」「单进程 → Nginx 反向代理 + PM2 集群」逐步演进。

### 1.2 分层架构（后端）

```
┌──────────────────────────────────────────────┐
│                Routes (HTTP)                 │  ← Fastify route handlers，参数校验（Zod）
├──────────────────────────────────────────────┤
│              Services (Business)             │  ← 业务编排，事务，跨资源协作
├──────────────────────────────────────────────┤
│         Repositories (Data Access)           │  ← Drizzle ORM 包装的数据访问
├──────────────────────────────────────────────┤
│   Infra: Storage / Auth / Image / Cron …     │  ← 基础设施抽象
└──────────────────────────────────────────────┘
```

> 分层规则：上层只能依赖下层；同层之间通过显式注入；不允许跨层（例如 Routes 直接调 Repositories）。

## 4. 核心流程详细设计

### 4.1 注册 / 登录（含 Refresh Token）

**Token 体系**：
- **Access Token**：短期 JWT，**15 分钟**有效，携带在 `Authorization: Bearer <token>` 头部，前端存于内存（Pinia）。
- **Refresh Token**：长期 JWT 或随机串，**30 天**有效，存于 **`httpOnly` + `Secure` + `SameSite=Lax` Cookie**，前端 JS 不可读（防 XSS）。
- DB 中维护 `refresh_tokens` 表（或在 `users` 表上记录最后一次签发的 jti），支持吊销。

**接口**：
```
POST /api/auth/register { email, password }
  └─ argon2 哈希 → INSERT users → 同登录流程下发双 token

POST /api/auth/login    { email, password }
  └─ argon2 verify
  └─ 签发 access (15m) + refresh (30d)
  └─ 响应体返回 access；refresh 通过 Set-Cookie 下发

POST /api/auth/refresh
  └─ 读取 cookie 中的 refresh token → 校验 + 查吊销表
  └─ 签发新的 access；可选择「rotate」即同时签发新的 refresh 替换旧的（推荐 ✅）
  └─ 旧 refresh 加入吊销表

POST /api/auth/logout
  └─ 吊销当前 refresh + 清除 cookie

GET  /api/auth/me   (Authorization: Bearer xxx)
```

**前端拦截器策略**：
- 请求拦截器：自动带上 access token。
- 响应拦截器：遇 401 + 错误码 `TOKEN_EXPIRED` → 自动调 `/api/auth/refresh` → 成功后重发原请求；失败则跳转登录页。
- 同时只允许一个刷新请求在途（用 Promise 队列），避免并发刷新打架。

> MVP 不做邮箱验证、不做找回密码（写在「后续优化」里）。

### 4.2 创建分享 + 上传图片

**MVP 采用「整体上传 + 并发分片请求」**（不做断点续传，作为后续优化）：

```
1. 前端：用户选图 → 选择有效期 → 选择上传模式（原图 / 压缩） → 点击「创建分享」
   POST /api/shares  { ttlSeconds }   (登录态)
     └─ 后端生成 shareId + shareCode + expiresAt → 返回
2. 前端：拿到 shareId 后，并发上传图片（每个文件一个请求，限制并发数=3）
   ┌─ 若选「压缩上传」：先用 browser-image-compression
   │   - 长边 ≤ 4000 px
   │   - JPEG q=0.92
   │   - 保留 EXIF（库默认行为）
   │   → 得到压缩后的 File 对象
   └─ 直接（或将压缩后的文件）走 multipart 上传
   POST /api/shares/:shareId/photos  (multipart/form-data, 单文件)
     └─ 后端：
         a. 流式落盘 storage/originals/<shareId>/<photoId>.<ext>
         b. sharp：
            - 生成 thumbnail（短边 ~400，**JPEG**，q=80）→ storage/previews/
            - 生成 medium   （长边 ~1600，**JPEG**，q=85）→ storage/mediums/
         c. 读取 width/height/size，写入 photos 表
         d. 返回 { photoId, width, height, size }
3. 前端：全部成功后，显示分享码 + 复制按钮
```

**关键决策**：
- 预览图**同步生成**（MVP 简单可靠；图片量大时再改异步队列）。
- **每文件单独请求** 而非一次 multipart 多文件，便于进度展示与失败重试。
- 单文件上限：MVP 设 **50MB**（可调），单分享照片数上限：**500 张**（可调）。
- **上传模式**（前端选项，与后端无关）：
  - `original`（默认）：不压缩，上传用户实际选择的文件。
  - `compressed`：前端用 `browser-image-compression` 压缩为 JPEG（长边≤4000，q=0.92）后再上传。
  - 「原图下载」下载的是**用户实际上传的那一份**（压缩则下到压缩版）。后端不区分两种模式，均视作「原图」存储。
  - DB 字段 `photos.uploaded_as`（`'original' | 'compressed'`）记录元数据，便于查看者了解（可选展示）。

### 4.3 凭码查看相册

```
GET /api/v/:code
  └─ 校验：分享存在 + 未过期 + 未手动结束
  └─ 返回 { shareId, expiresAt, photos: [{ id, width, height, thumbUrl, mediumUrl }] }

GET /api/v/:code/photos/:photoId/thumb     → 返回缩略图（webp）
GET /api/v/:code/photos/:photoId/medium    → 返回中等预览图（点开大图时加载）
GET /api/v/:code/photos/:photoId/original  → 返回原图（下载/最高清查看）
GET /api/v/:code/download                  → 流式 zip，包含全部原图
```

**关键决策**：
- 列表接口**只返回元数据**（含 thumbUrl/mediumUrl 占位 URL），不直接返回图片字节。
- 缩略图、中等图、原图**都通过 Fastify 流式返回**，`@fastify/static` 可选用于性能优化。
- 大图查看采用「先 medium 后 original」：PhotoSwipe 配 `srcset`，先快速展示中等图，需要"查看原图/下载"时再拉 original。

### 4.4 批量下载（zip 流）

```
GET /api/v/:code/download
  └─ 校验
  └─ res.headers: Content-Type: application/zip
                  Content-Disposition: attachment; filename="album-<code>.zip"
  └─ archiver('zip', { store: true })  // 不再二次压缩，节省 CPU
       .append(fs.createReadStream(originalPath), { name: filename })
       ... 多次 append
       .finalize() → pipe(reply.raw)
```

**关键点**：
- `store: true` —— 图片已经是压缩格式，再压几乎无收益还烧 CPU。
- 全程**流式**：边读原图边写到响应，不在内存堆 zip。
- 文件名冲突处理：以 `<photoId>-<originalName>` 命名。
- 支持中途断开：archiver 错误处理 → 关闭流。

### 4.5 续期 / 提前结束

```
PATCH /api/shares/:shareId/extend  { extendSeconds }   (登录态，且必须是创建者)
  └─ UPDATE shares SET expires_at = expires_at + ?
DELETE /api/shares/:shareId                            (登录态，且必须是创建者)
  └─ UPDATE shares SET status='ended', ended_at=now    （软结束）
  └─ 触发清理（同步删除文件 + 标记 cleaned_at）
```

> 续期上限可设（如最长不超过 30 天）；MVP 暂不限。

### 4.6 过期清理（cron）

```
node-cron: '*/1 * * * *'   // 每分钟扫一次
  ├─ SELECT shares WHERE
  │      (status='ended' OR (status='active' AND expires_at <= now))
  │      AND cleaned_at IS NULL
  ├─ 对每个 share：
  │     1. 删除 storage/originals/<shareId>/
  │     2. 删除 storage/previews/<shareId>/
  │     3. 删除 storage/mediums/<shareId>/
  │     4. DELETE photos WHERE share_id = ?
  │     5. UPDATE shares SET status='cleaned', cleaned_at=now
  └─ 失败：记日志 + 下一轮重试（不抛出导致 cron 中止）
```

**关键点**：
- 软删除 + 后台清理 → 用户可立即看到「已结束」，文件清理稍后异步完成，避免阻塞响应。
- 清理顺序：**先删文件，后改 DB**；中途失败下一轮可重试。
- 单次扫描限制处理数量（如 50 个/轮），避免单次跑太久。

## 5. 存储目录与生命周期

### 5.1 目录约定

```
storage/
├── originals/<shareId>/<photoId>.<ext>     # 原始格式（jpg/png/heic/...）
├── previews/<shareId>/<photoId>.jpg        # 缩略图：短边 ~400, JPEG q80
└── mediums/<shareId>/<photoId>.jpg         # 中等图：长边 ~1600, JPEG q85

data/
└── app.db                                   # SQLite 数据库
```

- `storage/` 与 `data/` 都加入 `.gitignore`。
- 路径常量集中在 `infra/storage/paths.ts`，禁止业务代码自己拼路径。

### 5.2 生命周期

| 状态 | 描述 | DB 字段 | 文件存在？ |
| --- | --- | --- | --- |
| `active` | 生效中 | `status='active'`, `expires_at>now` | 是 |
| `ended` | 用户主动结束（软） | `status='ended'` | 暂存（待 cron 清） |
| `expired` | 到期未清理（软） | `status='active'` 但 `expires_at<=now` | 暂存（待 cron 清） |
| `cleaned` | 已物理清理 | `status='cleaned'`, `cleaned_at` | 否 |

> 允许「分享码用户已不可访问，但记录还在」的短暂窗口（≤1 分钟），便于审计与日志。

## 7. 横切关注点

### 7.1 鉴权装饰器

```ts
// plugins/auth.plugin.ts
fastify.decorate('authenticate', async (req, reply) => { ... })

// 使用
fastify.get('/api/me/shares', { preHandler: [fastify.authenticate] }, handler)
```

### 7.2 限流策略（@fastify/rate-limit）

| 路由 | 限制 |
| --- | --- |
| `POST /api/auth/login` | 10 次 / 分钟 / IP |
| `GET /api/v/:code`（凭码访问） | 30 次 / 分钟 / IP |
| 其他公共接口 | 60 次 / 分钟 / IP |

### 7.3 错误处理

- 业务错误：抛 `BizError(code, message, httpStatus)`，统一插件转换为 `{ code, message }`。
- 校验错误：Zod 报错 → 400 + `code='VALIDATION_FAILED'` + 详情。
- 未捕获错误：兜底插件返回 500 + `code='INTERNAL_ERROR'`，记 pino error 日志。

### 7.4 日志

- pino，按级别：`info`（接入/启动）、`warn`（业务异常）、`error`（系统异常）。
- 请求日志使用 Fastify 内置 `requestId`，便于追踪。
- 文件操作（写入/删除）必须打 info 日志。
